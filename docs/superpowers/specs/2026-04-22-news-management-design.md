# 资讯管理优化设计

- 日期: 2026-04-22
- 项目: `my-blog`
- 范围: 后台资讯管理、RSS 抓取链路、前台资讯页缓存一致性
- 结论: 将资讯系统收敛为纯 RSS 聚合流程，去掉手动抓取和手动录入，补齐后台管理能力与数据一致性

## 背景

当前资讯系统已经具备基础的 RSS 来源、资讯入库、前台展示和后台管理能力，但整体仍处于半成品状态。

现状主要问题如下：

- 后台资讯列表只展示第一页数据，没有分页和来源筛选，资讯数量增长后无法有效维护。
- 后台支持手动抓取和手动录入，定时任务也会自动抓取，系统存在多条写入路径，职责不清晰。
- 前台 `/news` 页面使用 15 分钟 ISR，但资讯管理相关接口没有触发 `revalidatePath('/news')`，后台操作后前台经常延迟可见。
- `fetchFeed` 的抓取统计只有单个 `fetched` 计数，无法区分新增、更新、过滤和失败，且错误被吞掉，运营不可观测。
- 来源更新接口已支持修改 `name/feedUrl/siteUrl/active`，但后台页面没有对应交互，后端能力与前端能力脱节。
- 资讯展示依赖 `NewsItem.sourceName` 冗余字段，来源改名后旧资讯可能继续显示旧名字。
- 过期清理按 `createdAt` 执行，不符合资讯内容的时间语义。

## 目标

- 将资讯系统收敛为单一路径的 RSS 聚合流程。
- 去掉后台手动抓取和手动录入能力，后台只负责配置来源和管理已抓取内容。
- 补齐后台资讯列表的分页、筛选和状态展示能力。
- 提升抓取与清理逻辑的可观测性、容错性和语义正确性。
- 保证后台变更会立即同步到前台 `/news` 页面。
- 在不做破坏性迁移的前提下兼容已有旧数据。

## 非目标

- 本次不改造 RSS 来源的发现策略，不新增自动推荐来源能力。
- 本次不引入资讯分类、标签、批量编辑、批量删除、归档或隐藏等更重的内容治理能力。
- 本次不删除 `NewsItem.sourceName` 字段，也不执行数据迁移清理旧的 `manual` source。
- 本次不处理与资讯模块无关的现有 Markdown 测试失败。

## 设计概览

资讯系统调整为以下职责模型：

- 定时任务负责唯一的抓取入口。
- 后台 `资讯管理` 页面负责来源配置和已抓取资讯管理。
- 前台 `/news` 页面继续负责聚合展示，但依赖及时的缓存刷新。

系统中将不再存在以下后台能力：

- 手动抓取全部来源
- 手动抓取单个来源
- 手动添加单条资讯

## 数据流设计

### 抓取入口

抓取入口只保留定时任务。

- `instrumentation.ts` 中的 cron 任务继续调用抓取逻辑。
- 后台页面移除“抓取全部”和“抓取”按钮。
- `src/app/api/admin/fetch-news/route.ts` 删除。

这样做的原因：

- 系统只有一条写入路径，行为稳定且可预期。
- 后台管理页面职责更清晰，不再既是配置台又是操作面板。
- 抓取结果只需要围绕定时任务日志优化，不必维护额外的后台触发流程。

### 来源管理

后台来源管理保留以下能力：

- 新增 RSS 来源
- 编辑来源名称、Feed URL、站点 URL
- 启用或停用来源
- 删除来源

来源启停会直接影响定时抓取是否包含该来源。

### 资讯管理

后台资讯管理保留以下能力：

- 查看资讯列表
- 按来源筛选
- 翻页浏览
- 删除单条资讯

后台资讯管理移除以下能力：

- 手动添加资讯
- 通过后台触发抓取

## API 设计

### `GET /api/admin/news-sources`

保留，返回来源列表及关联资讯数量。

响应需要继续支持以下字段：

- `id`
- `name`
- `feedUrl`
- `siteUrl`
- `type`
- `active`
- `lastFetched`
- `_count.newsItems`

### `POST /api/admin/news-sources`

保留，用于新增 RSS 来源。

约束：

- `name` 必填
- `feedUrl` 必填
- `siteUrl` 可选
- `type` 默认 `rss`

### `PUT /api/admin/news-sources/[id]`

保留，用于来源编辑与启停。

支持更新：

- `name`
- `feedUrl`
- `siteUrl`
- `active`

成功后需要执行：

- `revalidatePath('/news')`

原因：

- 来源名变化需要立即反映到前台资讯卡片。
- 来源停用或启用会影响抓取后的前台可见内容。

### `DELETE /api/admin/news-sources/[id]`

保留，用于删除来源及其级联资讯。

成功后需要执行：

- `revalidatePath('/news')`

### `GET /api/admin/news-items`

保留，作为后台资讯列表接口。

支持：

- `page`
- `sourceId`

返回内容需要包含：

- 当前页资讯列表
- 分页信息
- 资讯关联来源名称
- 发布时间
- 抓取时间

资讯列表查询应优先返回关联来源名称，而不是单独依赖 `sourceName` 快照字段。

### `POST /api/admin/news-items`

删除。

原因：

- 本次明确移除手动录入资讯能力。
- 系统不再维护 `manual` source 的新增路径。

### `DELETE /api/admin/news-items/[id]`

保留。

成功后需要执行：

- `revalidatePath('/news')`

### `POST /api/admin/fetch-news`

删除。

原因：

- 本次明确移除后台手动抓取能力。

### `GET /api/news`

保留，作为前台资讯数据接口和未来筛选能力的基础。

需要保证来源名称的返回逻辑与后台一致，优先使用关联来源名称，关联缺失时才回退 `sourceName`。

## 抓取逻辑设计

### 结果结构

`fetchFeed` 返回结构从单一 `fetched` 扩展为可运营的统计结果。

每个来源至少返回：

- `source`
- `inserted`
- `updated`
- `filtered`
- `failed`
- `errorMessage`

语义定义：

- `inserted`: 本次新写入数据库的资讯数
- `updated`: 已存在资讯但本次刷新了标题、摘要、发布时间或抓取时间的数量
- `filtered`: 因 AI 关键词规则被过滤的数量
- `failed`: 单条处理失败数量
- `errorMessage`: 来源级别抓取失败的错误信息，没有错误时为 `null`

### 写入行为

`upsert` 行为保留，但结果统计必须准确区分新增和更新。

实现要求：

- 不再通过吞掉异常的方式继续执行。
- 单条写入异常要累加到 `failed`，并保留来源级别的成功结果。
- 来源级别抓取异常才写入 `errorMessage` 并结束该来源抓取。

### 来源抓取顺序

`fetchAllActiveSources` 改为并行抓取活跃 RSS 来源，不再串行执行。

约束：

- 需要使用受控并行而不是无限并发，避免来源过多时造成外部请求尖峰。
- 在当前来源数量较小的情况下，可采用简单的 `Promise.allSettled`，前提是每个来源的错误都能被单独收集，不会中断整体结果。

设计选择：

- 当前项目来源数量较少，本次采用并行抓取即可，不额外引入并发控制库。

### `lastFetched`

来源只在该来源抓取流程结束时更新 `lastFetched`。

语义：

- 表示最近一次完成抓取尝试的时间，而不是最近一次抓到新内容的时间。

## 清理策略

旧逻辑按 `createdAt` 清理过期资讯，不符合资讯时间语义。

新逻辑改为：

- 优先基于 `publishedAt` 判断是否过期
- 若 `publishedAt` 为空，则回退使用 `fetchedAt`

过期阈值维持现有的 7 天。

删除条件等价于：

- `publishedAt < cutoff`
- 或 `publishedAt` 为空且 `fetchedAt < cutoff`

这样可以避免因不同来源发布时间缺失或入库时间差异导致的不合理清理。

## 来源名称一致性

数据库中保留 `NewsItem.sourceName`，但不再将其作为展示主数据。

展示规则改为：

1. 若关联 `source` 存在，则展示 `source.name`
2. 若关联缺失，则回退展示 `sourceName`
3. 若两者都没有，则展示默认占位文案

这样可以立刻修复来源改名后的前后台不一致问题，同时避免本次做 schema 级迁移。

## 后台页面设计

后台 `资讯管理` 页面调整为两个主要区块：

- `RSS 源管理`
- `资讯列表`

### 页面顶部状态区

顶部不再提供抓取按钮，而是提供状态信息：

- 自动抓取说明，明确系统通过定时任务抓取
- 当前启用中的 RSS 来源数量
- 最近抓取时间摘要，基于所有来源的 `lastFetched`

目的：

- 管理员无需手动触发抓取，也能判断系统是否正常工作。

### RSS 源管理区

每个来源展示：

- 名称
- Feed URL
- 站点 URL
- 状态
- 最近抓取时间
- 资讯数量

每个来源支持的操作：

- `编辑`
- `启用/停用`
- `删除`

交互采用行内编辑而不是弹窗。

进入编辑态后可修改：

- 名称
- Feed URL
- 站点 URL

提供：

- `保存`
- `取消`

设计理由：

- 来源的数据结构简单，行内编辑成本更低，维护更直接。

### 资讯列表区

资讯列表支持：

- 来源筛选
- 分页
- 查看标题、摘要、来源、发布时间、抓取时间
- 删除单条资讯

表现要求：

- 标题仍可点击跳转原文
- 摘要保持截断显示
- 来源名展示遵循“优先关联来源名”的规则

### 空态设计

空态文案按情况区分：

- 没有任何 RSS 来源
- 存在来源但没有启用中的来源
- 启用中的来源尚未抓到内容
- 当前筛选条件下没有结果

目的：

- 降低管理员排查成本，避免“暂无资讯”这种无信息量提示。

### 删除确认

删除来源时保留明确确认：

- 删除来源会级联删除该来源下所有资讯

删除单条资讯时可保持轻确认或直接删除，具体以当前项目交互风格为准；本次优先保持实现简单，不引入额外弹窗系统。

## 前台资讯页设计

前台 `/news` 页面整体视觉结构保持不变，但数据读取逻辑需要与后台管理保持一致。

要求：

- 来源名称优先使用关联 `source.name`
- 删除、来源改名、来源删除等后台操作后可立即看到结果

缓存策略：

- 页面继续保留 ISR
- 所有影响资讯内容的后台写操作补 `revalidatePath('/news')`

这样可以保留前台性能优势，同时避免运营后台与前台数据脱节。

## 兼容策略

### 旧的 `manual` source

本次不做数据库清理。

兼容策略：

- 不再新增新的 `manual` source
- 旧的 `manual` source 若存在，后台来源列表仍可看到并删除
- 旧的手动资讯若存在，资讯列表仍可显示和删除

### 旧的 `sourceName`

字段保留，作为关联丢失时的回退展示值。

### 旧的管理入口

以下 UI 完全移除：

- 手动添加资讯表单
- 手动抓取全部按钮
- 单来源抓取按钮

## 日志与可观测性

定时任务日志需要基于新结果结构输出汇总信息。

至少输出：

- 总新增数
- 总更新数
- 总过滤数
- 总失败数
- 失败来源数量

来源级错误需要在日志中保留来源名与错误信息，便于后续在部署环境中排查。

## 验收标准

以下条件全部满足时，本次设计视为实现完成：

1. 后台页面不再出现手动抓取和手动添加资讯入口。
2. 后台来源支持新增、编辑、启停、删除。
3. 后台资讯列表支持分页和来源筛选。
4. 删除资讯后，前台 `/news` 立即同步结果。
5. 来源改名后，前后台展示一致，不再保留旧名称问题。
6. 定时抓取结果能区分新增、更新、过滤和失败。
7. 过期清理按 `publishedAt` 或 `fetchedAt` 生效，而不是按 `createdAt`。
8. 已有旧数据不会因为本次改造而不可见或不可删除。

## 实现顺序

建议实现顺序如下：

1. 收敛后端接口，删除手动抓取和手动录入路径。
2. 改造抓取逻辑的结果结构、并行执行和清理策略。
3. 为资讯相关写操作补充 `revalidatePath('/news')`。
4. 改造后台 `资讯管理` 页面，接入来源编辑、启停、分页和筛选。
5. 调整前后台来源名称展示逻辑。
6. 进行功能验证与回归检查。

## 风险与取舍

### 只保留定时抓取的取舍

去掉手动抓取后，管理员无法立即通过后台补抓内容。

本次接受这个取舍，原因是：

- 用户已经明确要去掉手动抓取功能。
- 系统职责会显著更清晰。
- 未来如果需要重新引入手动抓取，应作为单独需求重新设计，而不是保留当前半成品入口。

### 保留 `sourceName` 字段的取舍

本次不清理冗余字段，会保留一定的模型冗余。

本次接受这个取舍，原因是：

- 可以避免 schema 迁移和历史数据回填。
- 能以最小代价修复来源改名后的展示不一致问题。

### 并行抓取的取舍

并行抓取会让定时任务更快，但也会增加短时间内的外部请求并发。

本次接受这个取舍，原因是：

- 当前来源数量有限。
- 相比串行抓取导致的整体延迟，并行的收益更直接。
